## List All Bookings (Admin)

<api-endpoint method="GET" path="/api/v1/bookings" />

Retrieve all bookings with advanced filtering options. Only accessible by super admins.

### Authentication

<ParamField header="Authorization" type="string" required>
  Bearer token for authentication. Required role: **Admin**
</ParamField>

### Query Parameters

<ParamField query="status" type="string">
  Filter by booking status. Options: `pending`, `confirmed`, `rider_assigned`, `delivered`, `cancelled`, `rejected`
</ParamField>

<ParamField query="page" type="string">
  Page number for pagination (default: 1)
</ParamField>

<ParamField query="limit" type="string">
  Number of items per page (default: 10)
</ParamField>

<ParamField query="category" type="string">
  Filter by place category
</ParamField>

<ParamField query="query" type="string">
  Search query for booking details
</ParamField>

<ParamField query="from" type="string">
  Start date filter (ISO 8601 format)
</ParamField>

<ParamField query="to" type="string">
  End date filter (ISO 8601 format)
</ParamField>

<ParamField query="place" type="string">
  Filter by specific place ID
</ParamField>

### Example Request

```bash
GET /api/v1/bookings?status=pending&page=1&limit=20&from=2026-03-01&to=2026-03-31
```

## Update Booking Status

The following endpoints allow authorized users to transition bookings through different states.

### Confirm Booking

<api-endpoint method="PUT" path="/api/v1/bookings/:id/confirm-booking" />

Confirm a pending booking. Transitions status from `pending` to `confirmed`.

**Required roles:** Admin, Place Admin

<ParamField path="id" type="string" required>
  UUID of the booking to confirm
</ParamField>

### Deliver Booking

<api-endpoint method="PUT" path="/api/v1/bookings/:id/deliver-booking" />

Mark a booking as delivered. Transitions status to `delivered`.

**Required roles:** Admin, Place Admin

<ParamField path="id" type="string" required>
  UUID of the booking to mark as delivered
</ParamField>

### Cancel Booking

<api-endpoint method="PUT" path="/api/v1/bookings/:id/cancel-booking" />

Cancel a booking. Transitions status to `cancelled`.

**Required roles:** Admin, Place Admin

<ParamField path="id" type="string" required>
  UUID of the booking to cancel
</ParamField>

### Reject Booking

<api-endpoint method="PUT" path="/api/v1/bookings/:id/reject-booking" />

Reject a booking. Transitions status to `rejected`.

**Required roles:** Admin, Place Admin, User

<ParamField path="id" type="string" required>
  UUID of the booking to reject
</ParamField>

### Example Request

```bash
PUT /api/v1/bookings/a1b2c3d4-e5f6-7890-abcd-ef1234567890/confirm-booking
Authorization: Bearer <token>
```

### Example Response

```json
{
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": {
    "label": "confirmed"
  },
  "updated_at": "2026-03-04T11:00:00Z"
}
```

## State Transition Rules

- `pending` → Can transition to `confirmed`, `rejected`, or `cancelled`
- `confirmed` → Can transition to `rider_assigned`, `delivered`, or `cancelled`
- `rider_assigned` → Can transition to `delivered` or `cancelled`
- `delivered`, `rejected`, `cancelled` → Terminal states (no further transitions)

## Notes

- All booking ID parameters must be valid UUIDs
- Place admins can only manage bookings for their assigned place
- Users can only delete their own bookings
- Status transitions are validated server-side to ensure proper workflow
- Some endpoints may trigger notifications to relevant parties (users, place admins, riders)
